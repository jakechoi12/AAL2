## 1. 서비스 개요

### 1.1 한 줄 정의

**글로벌 시장/리포트/기업DB/거래데이터를 AI Agent가 선제적으로 분석해, 바이어·셀러 인사이트 → 소싱/영업 실행 → 결제 → 물류견적 → 수출입/통관/배송까지 사용자는 ‘컨펌’만 하는 B2B 실행형 플랫폼**

### 1.2 해결하려는 문제

* 정보가 흩어져 있음(리포트, 트렌드, 기업 DB, RFQ, 물류)
* B2B 거래는 “찾기→검증→연락→견적→계약→결제→물류/통관”이 길고 담당자 의존도가 큼
* 특히 해외 유통체인/바이어 발굴은 **초기 영업 비용이 높고** 반복 작업이 많음

### 1.3 목표 사용자

* **셀러** : 대기업/중견/중소 수출기업(가격 공개/비공개 혼재)
* **바이어** : 해외 유통체인/도매상/프로젝트 바이어
* **운영자(내부)** : 온보딩/검증/거래/리스크 관리(초기 성공의 핵심)

## 3. 제품 컨셉: “Intelligence → Insight → Execution” 파이프라인

### 3.1 핵심 사용자 경험(UX) 원칙

* 사용자는 **입력**이 아니라 **컨펌**만 한다.
* AI는 **선제적**으로 “기회/리스크/다음 액션”을 만든다.
* 커머스는 메뉴가 아니라 **AI 액션을 실행하는 엔진**이다.

### 3.2 End-to-End 흐름

1. 글로벌 마켓 리포트/뉴스/데이터 수집
2. 국가·지역 트렌드 분석 + 공급망(SC) 레벨까지 맵핑(가능 범위부터 단계적)
3. 바이어/셀러 인사이트 생성(누가/왜/언제/얼마나)
4. 관련 기업 안내(최초 판매자–중간 유통–최종 구매자까지 가치사슬 상 위치)
5. 영업/마케팅 방법 제안(채널, 메시지, 타깃, 접촉 순서)
6. RFQ 생성/배포 또는 카탈로그 구매 유도
7. 결제
8. 물류 견적/스케줄 추천
9. 실제 수출입, 통관, 배송 진행(트래킹/문서/상태)

## 4. 핵심 기능 구성

### 4.1 Intelligence 모듈

* **News Intelligence** : 규제/수요/공급/리스크 신호 감지
* **Market Data Intelligence** : 가격/운임/환율/수요 지표 변화 감지
* **Report Intelligence** : 리포트 자동 요약/키포인트/기회국가 추천
* **SC Trend 분석(단계적)** : 완제품→부품/소재→공급처/수요처 연결 (초기엔 “카테고리/산업” 중심으로 시작 후 확장)

### 4.2 Directory 모듈 (Kompass/KTradeHub 레퍼런스)

* **기업 DB(바이어/셀러)** : 국가/산업/카테고리/인증/규모/거래 방식 태깅
* **Value Chain View(옵션)** : 최초 판매자–유통–최종 구매자(기업 역할을 “제조/브랜드/유통/리테일”로 라벨링)

### 4.3 Commerce 모듈 (B2B몰 + RFQ 혼합)

* **카테고리 허브**
  * 구매자 리스트
  * 판매자 리스트
  * 공개 RFQ 보드
* **가격 정책 혼합**
  * 공개 가격(중소/중견 중심)
  * 범위 가격(From~)
  * 비공개(RFQ Only, 대기업 선호)
* **RFQ**
  * 공개 RFQ(마켓)
  * 비공개 RFQ(초청/선정 셀러 대상)

### 4.4 Agent 모듈 (Rufus 레퍼런스 확장)

* **대화형 AI Agent**가 아래를 수행
  * “이 국가에서 이 제품 팔려면?” → 트렌드+리스크+추천 바이어+추천 셀러+가격 범위+물류까지 제안
  * RFQ 자동 작성/배포
  * 셀러 응답 견적 초안 생성
  * 바이어별 영업 메시지/메일 초안 생성
  * 견적 비교/추천(가격+납기+리스크+물류)
  * 사용자는 Confirm/Modify/Reject만

### 4.5 결제/정산 모듈

* 초기: **거래 조건 확정 + 인보이스/계약 문서 관리 + 결제 요청** 중심
* 확장: 에스크로/신용장/분할 결제 등(국가별 정책/리스크 기반)

### 4.6 물류/통관/배송 모듈 (팬스타/산스타 강점)

* 포워더 견적 요청/비교/추천
* 선적 스케줄 추천
* 통관 서류 체크리스트/생성 지원
* 트래킹/상태 관리(견적→예약→선적→통관→배송)

## 5. 핵심 플로우(시나리오) 3종

### 시나리오 A: “시장 기회 감지 → RFQ 자동 발행”

1. AI가 특정 국가/카테고리 수요 증가 + 운임 하락 감지
2. “이 타이밍에 RFQ 발행” 카드 제안
3. 사용자가 컨펌
4. 공개 RFQ 게시 + 추천 셀러/바이어 자동 매칭

### 시나리오 B: “공개 RFQ 발견 → 셀러 응답 자동화”

1. 공개 RFQ 신규 등록 감지
2. 카테고리/인증 기반으로 셀러 추천
3. 응답 견적 초안 생성(공개/비공개 옵션 포함)
4. 셀러 컨펌 후 제출

### 시나리오 C: “거래 확정 → 물류/통관 실행”

1. 최종 조건 확정
2. AI가 최적 포워더/스케줄 제안
3. 컨펌
4. 예약/서류/통관/트래킹 진행

## 6. MVP 범위 제안 (현실적인 1차 출시)

### MVP 목표

**“AI가 거래 시작(RFQ/매칭/견적요청)을 대신 해주는 경험”을 증명**
(결제/통관 완전 자동화는 2단계로)

### MVP 기능 묶음

1. 카테고리 허브(구매자/판매자 리스트 + 공개 RFQ 보드)
2. RFQ(공개/비공개) + 셀러 매칭 + 견적 수신
3. 가격 정책(공개/범위/비공개) 혼합
4. AI Agent:
   * (a) 기회 감지 → RFQ 초안 생성
   * (b) 견적 비교/추천 카드
   * (c) 영업 메시지/이메일 초안 생성
5. 물류: 포워더 견적 요청/비교(이미 프로토타입 기반 확장)

## 1. 유즈케이스 한 줄 요약 (임원/외부 설명용)

> **“AI가 글로벌 시장·공급망 신호를 분석해
> 도레이첨단소재의 수출 기회를 선제적으로 포착하고,
> 바이어 발굴 → RFQ → 견적 → 물류 실행까지
> 담당자는 ‘컨펌’만 하는 B2B 수출 자동화 시나리오”**

---

## 2. 문제 상황 (현실)

### 도레이첨단소재 관점

- 동남아/인도 전자·자동차 산업 성장 중
- 하지만:
  - 어느 국가/고객이 **지금** 유효한지 판단이 늦음
  - RFQ 대응은 인력 소모가 큼
  - 물류 조건(운임/리드타임)이 영업 판단에 즉시 반영되지 않음

👉 **“정보는 많은데, 실행까지 이어지지 않는다”**

## 3. AI Agent가 개입하는 전체 흐름

### 전체 플로우 요약

```
시장/뉴스/리포트
      ↓
AI 인텔리전스
(수요·공급·SC 분석)
      ↓
수출 기회 탐지
      ↓
AI Action 제안
(RFQ / 바이어 / 물류)
      ↓
사용자 컨펌
      ↓
거래·물류 실행
```

## 4. 단계별 상세 유즈케이스

---

### STEP 1. 글로벌 마켓 & 공급망 신호 감지 (AI Thinking)

#### 입력 데이터

- 글로벌 마켓 리포트
  - 동남아 전자제품 생산량 증가
- 뉴스 인텔리전스
  - 베트남/태국 반도체·전자 투자 확대
- 마켓 데이터
  - 기능성 필름 가격 안정
  - 해상 운임 하락(NEA → SEA)

#### AI 인사이트 생성

> “동남아 전자 산업 성장 + 소재 수요 증가 + 물류비 하락
> → **첨단소재 수출 타이밍 적합**”

---

### STEP 2. 산업 기반 SC 트렌드 분석 (초기 버전)

AI가 완제품 관점에서 역으로 분석:

```
전자 완제품
 → 디스플레이/모듈
   → 기능성 필름 / 엔지니어링 플라스틱
     → 도레이첨단소재 주요 제품군
```

#### AI 출력

- **유망 제품군**
  - 고기능성 필름
  - 엔지니어링 플라스틱(PPS 등)
- **유망 국가**
  - 베트남, 태국, 인도
- **구매 주체**
  - 지역 유통체인
  - Tier1 제조사

---

### STEP 3. 바이어·셀러 인사이트 자동 생성

#### 바이어 인사이트

- 해당 국가 전자 부품 유통체인 리스트
- 구매 패턴
  - RFQ 중심
  - MOQ 높음
- 거래 선호 조건
  - FOB/CIF 혼용
  - 리드타임 중요

#### 셀러 포지션

- 도레이첨단소재:
  - Enterprise Seller
  - 가격 비공개(RFQ only)
  - 품질/납기 강점

---

### STEP 4. AI Action Agent 제안 (핵심 장면)

#### AI가 사용자에게 제안하는 카드

> **[Action 제안]
> “도레이첨단소재 – 동남아 전자소재 RFQ 발행”**
>
> - 대상 국가: 베트남 / 태국
> - 추천 바이어: 검증된 유통체인 3곳
> - 제품군: 기능성 필름
> - 가격 정책: 비공개 RFQ
> - 예상 물류 리드타임: 12~15일
> - 추천 출항: 2주 내

👉 버튼:

- ✅ RFQ 발행
- ✏️ 조건 수정
- ❌ 보류

---

### STEP 5. 사용자 컨펌 (Human-in-the-loop)

도레이 담당자는:

- 제품군 확인
- 수량 범위 조정
- 국가 1곳 추가

👉 **Confirm 클릭**

---

### STEP 6. RFQ 실행 & 셀러-바이어 매칭

#### 플랫폼이 자동으로 수행

- RFQ 초안 확정
- 대상 바이어에게 배포
- RFQ 상태 트래킹

#### 바이어 측

- RFQ 수신
- 플랫폼을 통해 문의/응답

---

### STEP 7. 견적 수신 & 비교 (AI 지원)

- 바이어 조건별 요구사항 정리
- AI가:
  - 가격 조건
  - 납기
  - 물류 리스크
  - 예상 마진
    을 비교 요약

👉 “이 바이어 조건이 가장 유리합니다” 카드 제시

---

### STEP 8. 물류 견적 & 실행 (팬스타 강점)

#### AI 물류 에이전트

- 팬스타/포워더 견적 병렬 요청
- 운임/스케줄 비교
- 최적 루트 추천

도레이 담당자:

- **컨펌만 진행**

---

### STEP 9. 실제 수출입/통관/배송

- 선적 예약
- 통관 서류 체크리스트
- 배송 트래킹
- 상태 리포트 자동 생성

## 1. 유즈케이스 한 줄 요약 (바이어 관점)

> **“AI가 글로벌 시장·공급망을 분석해
> 한국 바이어에게 ‘지금 가장 유리한 해외 공급처’를 추천하고,
> 소싱 → RFQ → 견적 → 결제 → 수입·통관·배송까지
> 담당자는 컨펌만 하는 글로벌 조달 자동화 시나리오”**

---

## 2. 문제 상황 (한국 바이어 현실)

### 한국 바이어의 공통 Pain Point

* 원가 압박은 심한데:
  * 해외 신규 공급처 발굴이 느림
  * 기존 거래처 의존도가 높음
  * 공급 리스크(납기/운임/정치·규제)를 실시간으로 반영하기 어려움
* 소싱/구매 담당자는:
  * 리포트 따로
  * 기업 DB 따로
  * RFQ/이메일 따로
  * 물류는 또 다른 시스템

👉 **“정보는 넘치는데, 결정과 실행은 여전히 수작업”**

> **대체 공급처를 지금 확보하는 것이 유리** ”

### STEP 2. 완제품 기준 공급망 역분석

AI가 한국 바이어의 완제품 기준으로 분석:

```
완제품(전자부품)
 → 핵심 소재(기능성 필름)
   → 해외 제조사
     → 물류 리스크

#### AI 출력

* 대체 가능 소재 카테고리
* 필수 스펙
* 품질/인증 요구사항

### STEP 3. 글로벌 셀러 인사이트 생성

#### AI 추천 셀러 풀

* 일본(고품질, 고가)
* 베트남/태국(중간 가격, 납기 유리)
* 인도(가격 경쟁력, MOQ 높음)

#### 각 셀러별 요약

* 가격 정책(공개/비공개)
* MOQ/리드타임
* 거래 선호 Incoterms
* 과거 거래 패턴

### STEP 4. AI Action Agent 제안 (핵심 장면)

> **[Action 제안]

> * 셀러 유형: 제조사 직접

> * 추천 물류 조건: CIF Busan
>
> 👉 진행하시겠습니까?

버튼:

* ✅ RFQ 발행
* ✏️ 조건 수정
* ❌ 보류

* 납기
* MOQ
* 환율 영향
* 물류 리스크

#### AI 요약 카드

> “베트남 A사는 단가 최저,

## 15. RFQ 프로세스 상세 설계

### 15.1 RFQ 유형 정의

| 유형                          | 발행자 | 목적                | 예시                             |
| ----------------------------- | ------ | ------------------- | -------------------------------- |
| **구매 RFQ (Buy RFQ)**  | 바이어 | 상품/소재 구매      | "기능성 필름 10,000kg 구매 희망" |
| **판매 RFQ (Sell RFQ)** | 셀러   | 재고/생산 물량 판매 | "PPS 50,000kg 즉시 출하 가능"    |

| 공개 범위         | 설명                  | 대상                |
| ----------------- | --------------------- | ------------------- |
| **PUBLIC**  | 모든 사용자에게 공개  | 전체 셀러/바이어    |
| **PRIVATE** | 초대된 기업만 열람    | 지정 기업           |
| **INVITED** | 추천 기업 + 직접 초대 | AI 추천 + 수동 초대 |

### 15.2 RFQ 상태 정의 및 상세 흐름도

#### 15.2.1 RFQ 상태 정의

| 상태             | 코드          | 설명                  | 진입 조건               |
| ---------------- | ------------- | --------------------- | ----------------------- |
| **초안**   | `DRAFT`     | 작성 중, 미게시       | 최초 생성 시            |
| **게시됨** | `OPEN`      | 공개되어 견적 수신 중 | 발행 버튼 클릭          |
| **마감됨** | `CLOSED`    | 견적 수신 종료        | 마감일 도래 / 수동 마감 |
| **확정됨** | `AWARDED`   | 특정 견적 선택 완료   | 견적 수락               |
| **완료됨** | `COMPLETED` | 거래까지 완료         | Transaction 완료        |
| **취소됨** | `CANCELLED` | 발행자가 취소         | 취소 요청               |
| **만료됨** | `EXPIRED`   | 응답 없이 마감        | 마감 + 견적 0건         |

### 15.3 견적(Quotation) 상태 정의 및 흐름도

#### 15.3.1 견적 상태 정의

| 상태             | 코드            | 설명                 | 진입 조건      |
| ---------------- | --------------- | -------------------- | -------------- |
| **초안**   | `DRAFT`       | 작성 중              | 최초 생성      |
| **제출됨** | `SUBMITTED`   | 바이어에게 전송 완료 | 제출 버튼 클릭 |
| **열람됨** | `VIEWED`      | 바이어가 확인        | 바이어 열람 시 |
| **협상중** | `NEGOTIATING` | 조건 협의 진행       | 수정 요청 발생 |
| **수락됨** | `ACCEPTED`    | 바이어가 선택        | 견적 수락      |
| **거절됨** | `REJECTED`    | 바이어가 거절        | 견적 거절      |
| **만료됨** | `EXPIRED`     | 유효기간 경과        | 유효일 초과    |
| **철회됨** | `WITHDRAWN`   | 셀러가 철회          | 철회 요청      |

### 15.5 SLA (Service Level Agreement) 정책

#### 15.5.1 응답 시간 SLA

| 구분                | 대상           | 기준 시간   | 측정 방법                   |
| ------------------- | -------------- | ----------- | --------------------------- |
| **RFQ 열람**  | 초대받은 셀러  | 24시간 이내 | 초대 시점 → 첫 열람        |
| **견적 제출** | 견적 응답 셀러 | 72시간 이내 | RFQ 열람 → 견적 제출       |
| **견적 검토** | 바이어         | 48시간 이내 | 견적 수신 → 열람           |
| **견적 응답** | 바이어         | 7일 이내    | 견적 열람 → 수락/거절/협상 |
| **협상 응답** | 양측           | 48시간 이내 | 수정 요청 → 응답           |

#### 15.5.2 SLA 등급 및 혜택

| 등급               | 조건                                    | 혜택                      |
| ------------------ | --------------------------------------- | ------------------------- |
| **Gold**     | 응답률 95%+ / 평균 응답 24시간 이내     | 검색 상위 노출, Gold 배지 |
| **Silver**   | 응답률 85%+ / 평균 응답 48시간 이내     | 검색 우선 노출            |
| **Standard** | 응답률 70%+ / 평균 응답 72시간 이내     | 기본 노출                 |
| **Warning**  | 응답률 70% 미만 / 평균 응답 72시간 초과 | 노출 제한, 경고           |

#### 15.5.3 SLA 위반 시 페널티

| 위반 유형                       | 페널티                       |
| ------------------------------- | ---------------------------- |
| **무응답 (3회 연속)**     | RFQ 매칭 일시 중지 (7일)     |
| **무응답 (5회 연속)**     | 계정 경고 + 노출 제한 (30일) |
| **견적 철회 (잦은 발생)** | 신뢰도 스코어 감점           |
| **허위 견적**             | 계정 정지 + 운영자 검토      |
| **연락 두절**             | 계정 비활성화                |

### 15.6 자동화 규칙 (Automation Rules)

#### 15.6.1 RFQ 관련 자동화

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        RFQ 자동화 규칙 (Automation Rules)                   │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │ RULE 1: RFQ 발행 시 자동 매칭                                       │   │
│  ├─────────────────────────────────────────────────────────────────────┤   │
│  │ WHEN: RFQ.status = OPEN                                             │   │
│  │ THEN:                                                               │   │
│  │   1. AI가 카테고리/국가/인증 기준으로 셀러 매칭                    │   │
│  │   2. 매칭 셀러에게 알림 발송 (이메일 + 푸시)                       │   │
│  │   3. 매칭 결과를 AI_Action 테이블에 기록                           │   │
│  │   4. 바이어에게 "N개 셀러에게 전달됨" 알림                         │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │ RULE 2: 마감일 자동 상태 변경                                       │   │
│  ├─────────────────────────────────────────────────────────────────────┤   │
│  │ WHEN: NOW() >= RFQ.deadline AND RFQ.status = OPEN                   │   │
│  │ THEN:                                                               │   │
│  │   IF quotation_count > 0:                                           │   │
│  │     RFQ.status = CLOSED                                             │   │
│  │     NOTIFY 바이어: "RFQ 마감됨, 견적 검토 필요"                     │   │
│  │   ELSE:                                                             │   │
│  │     RFQ.status = EXPIRED                                            │   │
│  │     NOTIFY 바이어: "RFQ 만료됨 (응답 없음)"                         │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │ RULE 3: 마감 임박 리마인더                                          │   │
│  ├─────────────────────────────────────────────────────────────────────┤   │
│  │ WHEN: RFQ.deadline - NOW() <= 24시간 AND RFQ.status = OPEN          │   │
│  │ THEN:                                                               │   │
│  │   1. 미응답 셀러에게 리마인더 발송                                  │   │
│  │   2. 바이어에게 "마감 임박" 알림                                    │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │ RULE 4: 미열람 RFQ 리마인더                                         │   │
│  ├─────────────────────────────────────────────────────────────────────┤   │
│  │ WHEN: RFQ_Invitation.status = PENDING                               │   │
│  │       AND NOW() - invited_at >= 24시간                              │   │
│  │ THEN:                                                               │   │
│  │   NOTIFY 셀러: "새 RFQ가 대기 중입니다 (D-N일)"                    │   │
│  │   반복: 24시간 간격으로 최대 3회                                    │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

#### 15.6.2 견적 관련 자동화

```
┌─────────────────────────────────────────────────────────────────────────────┐

│  │ WHEN: Quotation.status = SUBMITTED                                  │   │

│                                                                             │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │ RULE 6: 견적 열람 시 상태 변경                                      │   │
│  ├─────────────────────────────────────────────────────────────────────┤   │
│  │ WHEN: 바이어가 Quotation 상세 페이지 접근                           │   │

│  ├─────────────────────────────────────────────────────────────────────┤   │

│  │       AND Quotation.status IN (SUBMITTED, VIEWED, NEGOTIATING)      │   │
│  │ THEN:                                                               │   │
│  │   1. Quotation.status = EXPIRED                                     │   │
│  │   2. 바이어에게 "견적 만료됨" 알림                                  │   │
│  │   3. 셀러에게 "견적이 만료되었습니다" 알림                          │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  ┌─────────────────────────────────────────────────────────────────────┐   │

│  │ THEN:                                                               │   │
│  │   1. Transaction 레코드 생성                                        │   │

│  │   3. 다른 견적들 status = REJECTED (자동)                           │   │
│  │   4. 바이어/셀러에게 "거래 시작" 알림                               │   │

│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │ RULE 9: 미검토 견적 리마인더                                        │   │
│  ├─────────────────────────────────────────────────────────────────────┤   │

### 17.2 수익원별 상세 설계

#### 17.2.1 거래 수수료 (Transaction Fee)

> 거래가 성사되어 결제가 완료될 때 부과

| 거래 금액 구간                     | 수수료율 | 최소 수수료 | 최대 수수료 |
| ---------------------------------- | -------- | ----------- | ----------- |
| $0 ~ $10,000                       | 2.5%     | $50         | -           |
| $10,001 ~ $50,000                  | 2.0%     | -           | -           |
| $50,001 ~ $100,000                 | 1.5%     | -           | -           |
| $100,001 ~ $500,000                | 1.0%     | -           | -           |
| $500,001 이상 | 0.8% | - | $10,000 |          |             |             |

**수수료 부과 대상:**

| 대상   | 부과 여부 | 비고             |
| ------ | --------- | ---------------- |
| 셀러   | ✅ 부과   | 판매 금액 기준   |
| 바이어 | ❌ 미부과 | 바이어 유입 유도 |

**수수료 예시:**

```
거래 금액: $125,000
구간: $100,001 ~ $500,000 (1.0%)

```
